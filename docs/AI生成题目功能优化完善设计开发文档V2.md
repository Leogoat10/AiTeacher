# AI生成题目功能优化完善设计开发文档 V2（聚焦版）

## 1. 文档目标

本版仅聚焦以下三类优化，避免过度工程化：
- 多轮对话体验优化
- AI上下文关联优化（最多关联最近5轮）
- Prompt优化（提升题目稳定性与可用性）

说明：
- 题目答案加密/隐藏逻辑保持现状，不在本次改造范围内。
- 暂不引入重试机制、密钥治理等平台化改造。

## 2. 当前实现现状（与本次目标相关）

### 2.1 对话与上下文
当前题目生成主流程是：
1. 前端提交表单参数（科目、年级、难度、题型、题量、补充要求）
2. 后端创建生成任务
3. 后端直接用本次参数拼接 Prompt 调用模型

问题：
- 当前生成任务基本是“单轮请求”，没有稳定利用历史对话语义。
- 用户在连续微调需求（例如“再难一点”“改成应用题”）时，模型不能持续理解上下文。
- 前端虽然有会话历史展示，但后端未将历史轮次作为结构化上下文参与下一次生成。

### 2.2 Prompt
当前 Prompt 为固定模板 + 本次表单参数，存在：
- 约束粒度不够细（题目分布、输出稳定性、去重策略表达弱）
- 用户补充要求与基础参数可能冲突，缺少优先级规则
- 对“多轮修订”场景缺少明确指令（例如“基于上一版仅调整难度，不改题型”）

### 2.3 前端交互
当前教师端支持历史会话切换和任务状态展示，但缺少：
- “本次是否带上最近上下文”的显式控制
- 多轮指令的可视化摘要（模型到底参考了哪几轮）

## 3. 优化目标

本次优化要达成：
1. **支持最多5轮上下文关联**：每次生成时自动携带最近5轮有效问答摘要。
2. **多轮意图连续**：支持“在上一轮基础上微调”的自然交互。
3. **Prompt可控**：模板化拆分系统约束、任务参数、历史上下文、本轮增量要求。
4. **可解释**：在前端可看到“本次参考了哪些历史轮次（简述）”。

## 4. 设计方案

## 4.1 上下文关联（5轮）

### 4.1.1 上下文选择规则
每次创建新生成任务时，后端按如下规则拼接上下文：
- 仅取当前会话内最近5轮（用户+AI为一轮）
- 过滤无效轮次：
  - 纯失败提示（如“生成失败”）不纳入
  - 过长原文不直接拼接，先摘要
- 每轮保留结构：
  - 用户意图摘要（1行）
  - AI产出摘要（题型/难度/题量/主要知识点）

### 4.1.2 上下文注入位置
Prompt结构调整为：
1. System角色：题目生成统一规则
2. Context段：最近5轮摘要（结构化）
3. Current Request段：本轮表单参数 + 本轮补充要求
4. Output Schema段：严格JSON格式

### 4.1.3 冲突处理
当“历史上下文”与“本轮参数”冲突时，优先级：
1. 本轮明确参数（最高）
2. 本轮补充文本要求
3. 历史5轮上下文（最低）

并在Prompt中明确写入该优先级，减少模型漂移。

## 4.2 多轮对话优化

### 4.2.1 后端能力
新增“对话上下文构建器”（建议在当前服务内先以私有方法实现）：
- `buildRecentContext(conversationId, limit=5)`
- 输出统一文本块：`Round1...RoundN`

同时新增任务入参字段（可选）：
- `useContext: true/false`（默认 true）
- `contextRounds: 5`（默认 5，最大 5）

### 4.2.2 前端能力
教师端新增两个轻量选项：
- `关联最近上下文`（开关，默认开）
- `关联轮数`（固定显示5，可后续扩展）

在生成结果下方显示：
- “本次已关联最近5轮上下文”
- 或“本次未关联历史上下文（纯参数生成）”

## 4.3 Prompt优化（核心）

### 4.3.1 Prompt模板拆分
将当前拼接方式拆成4段模板：
- `SYSTEM_RULES_TEMPLATE`
- `CONTEXT_TEMPLATE`
- `CURRENT_REQUEST_TEMPLATE`
- `OUTPUT_JSON_TEMPLATE`

收益：
- 可维护
- 方便单独迭代“系统约束”与“输出格式”
- 多轮场景更稳定

### 4.3.2 系统规则建议（保守可落地）
系统规则明确：
- 必须输出合法JSON，不输出解释性前缀
- 题量必须与请求一致
- 每题必须包含：题干、答案、解析、题型
- 同一批次题目避免重复考点表述
- 若参考上下文，仅“继承意图”，不要复制原题

### 4.3.3 本轮增量意图模板
为多轮修订场景加入标准指令：
- “若用户表达为‘再难一些/再简单一些’，仅调整难度，不改变题型和题量（除非本轮参数明确修改）”
- “若用户表达为‘换题型’，保留科目与年级，按新题型重生成”

## 5. 数据与接口改动（最小化）

## 5.1 接口入参调整
`POST /teacher/question/v2/tasks` 增加可选字段：
- `useContext`：Boolean，默认 `true`
- `contextRounds`：Number，默认 `5`，最大 `5`

不改已有字段，保持兼容。

## 5.2 消息查询增强（建议）
为便于构造上下文，查询消息时建议可拿到：
- `question`
- `answer`
- `structured_status`（用于过滤失败轮次）

说明：不要求重构数据库，只做最小查询字段补充。

## 6. 开发任务排期（2周）

| 周次 | 目标 | 任务 | 产出 | 验收 |
|---|---|---|---|---|
| 第1周 | 上下文关联可用 | 1) 后端实现最近5轮上下文构建；2) 创建任务接口支持`useContext/contextRounds`；3) Prompt改为四段式模板 | 后端可按开关关联5轮上下文 | 连续3次“微调生成”时，结果能体现连续性 |

## 7. 验收用例（建议直接执行）

### 用例1：连续微调难度
1. 先生成“中等难度5道选择题”
2. 再输入“保持题型和题量，再难一些”
3. 再输入“再简单一些”

预期：
- 每轮都保持题型与题量一致
- 难度变化方向符合指令

### 用例2：切换题型
1. 生成“初中物理 5道选择题”
2. 下一轮输入“改成简答题，其他不变”

预期：
- 题型切换为简答题
- 科目、年级、题量保持一致

### 用例3：关闭上下文
1. 打开会话后先生成一轮
2. 关闭“关联最近上下文”
3. 再生成同参数但不同补充要求

预期：
- 第二次生成不受上一轮影响
- 前端明确显示“未关联历史上下文”

## 8. 风险与边界

- 若历史轮次文本过长，摘要质量会影响上下文效果。
- 若用户指令模糊（如“按刚才那个来”），仍可能出现偏差，需要前端提示用户补充明确参数。
- 本版不做重试与平台治理，极端网络抖动下仍可能失败，但不影响本次“多轮与Prompt”目标。
