> 文档类型：功能优化与研发设计（仅设计，不开发）  
> 版本：v2.0

---

## 1. 文档目标与范围

### 1.1 目标
在现有 AI 出题能力基础上，形成一套可执行、可分期上线、可量化验收的优化设计方案，使功能从“可生成”升级为“可控生成 + 可复用沉淀 + 可闭环反馈”。

### 1.2 范围
- 覆盖教师端出题、历史会话、题目分发、学生作答与AI评估相关链路。
- 输出架构、数据、接口、流程、风控、性能、实施计划。
- 本文档不包含代码开发和数据库实际变更执行。

### 1.3 非目标
- 本阶段不实现复杂算法（如深度知识图谱推理、协同过滤生产化）。
- 不调整现有登录鉴权体系核心框架。

## 2. 现状实现分析（基于当前代码）

## 2.1 已实现能力（事实）

### 2.1.1 教师 AI 出题
- 前端页：`frontend/src/views/teacher/teacherquestion.vue`
- 后端接口：`POST /teacher/question`
- 服务：`TeachingPlanQueServiceImpl.generateTeachingQuestion`
- 参数：科目、年级、难度、题型、题量、自定义要求、conversationId
- 模型调用：DeepSeek Chat（`messages` 当前仅传 1 条 user 消息）
- 结果存储：`messages(question, answer, conversation_id)`

### 2.1.2 对话管理
- 新建会话：`POST /teacher/newConversation`
- 历史列表：`GET /teacher/conversations`
- 会话详情：`GET /teacher/conversation/{conversationId}`
- 删除会话：`POST /teacher/deleteConversation`

### 2.1.3 展示与编辑体验（前端）
- Markdown 渲染（`marked`）
- KaTeX 公式渲染 + DOMPurify 安全净化
- 填空下划线占位保护
- AI消息多选、全选

### 2.1.4 导出与分发
- 导出：前端本地拼接文本，下载 `.doc`（非后端模板化导出）
- 分发：`POST /assignment/sendToCourse`，按选中 AI 内容逐条创建 assignment 并分发给课程学生

### 2.1.5 学生作答与AI评分
- 学生提交答案后，调用 DeepSeek 进行评分与分析
- 保存 `student_answers(ai_score, ai_analysis)`
- 教师可查看和修改评分分析

## 2.2 当前链路（真实）
1. 教师填写出题参数并提交。
2. 后端拼装文本 Prompt，调用 DeepSeek 单轮生成。
3. 将“标题（question）+ AI结果（answer）”存入 `messages`。
4. 教师在聊天页选择 AI 内容，可导出 `.doc` 或发送到课程。
5. 学生收到 assignment 后提交答案，系统调用 AI 批改并存档。

## 2.3 关键问题与根因

### 2.3.1 生成链路问题
- “会话存在，但生成未使用历史上下文”，本质仍为单轮生成。
- Prompt 以自由文本为主，缺乏结构化输出约束，导致结果格式波动。
- 未做生成结果的结构化解析与质量校验，错误题目直接进入后续环节。

### 2.3.2 数据模型问题
- `messages.question` 当前存的是“标题”，不是用户真实请求；追溯性弱。
- 缺少“题目实体层（question_bank）”，无法形成题库资产与复用闭环。
- assignment 直接持有大段 content，缺少标准化题目粒度。

### 2.3.3 体验与性能问题
- 同步调用，用户等待显著；无任务态、无进度、无可取消。
- 导出为前端简化 `.doc`，样式与结构一致性弱。
- 历史对话与题目资产没有有效分层（“会话历史”不等于“题库管理”）。

### 2.3.4 运营与治理问题
- 缺少可观测指标（成功率、平均时延、重试率、质量评分）。
- 无敏感内容/越权内容的双重防护链路（Prompt + 输出审查）。
- AI 评分解析依赖文本规则，稳定性受模型输出格式影响。

## 3. 优化目标与成功标准

## 3.1 产品目标
- 支持“多轮可控生成”：基于最近上下文进行追问改题。
- 建立“题目资产层”：生成结果可结构化入库、检索、复用、组卷。
- 打通“出题-分发-作答-反馈”闭环，形成教学迭代依据。

## 3.2 技术目标
- 同步改异步：提交即返回任务ID，支持进度与状态查询。
- 结构化输出：统一 JSON Schema，失败可重试与降级。
- 质量闸门：格式校验、规则校验、风险校验三段式。

## 3.3 量化指标（验收）
- 生成成功率 >= 98%
- 结构化解析成功率 >= 95%
- 首次响应时间（任务已受理）<= 1s
- 10题生成P95完成时长 <= 12s
- 教师二次编辑率下降 >= 30%

## 4. 总体优化架构

## 4.1 分层架构
- 会话层：Conversation / Message（保留）
- 任务层：GenerationTask（新增，异步状态机）
- 题目层：QuestionBank / QuestionItem / KnowledgeTag（新增）
- 作业层：Assignment / StudentAnswer（复用）
- 评估层：QualityCheck / FeedbackLoop（新增）

## 4.2 设计原则
- 先兼容后替换：保留现有接口，新增 v2 接口渐进切换。
- 先结构化后智能化：先确保输出稳定，再做推荐与自适应。
- 可观测优先：每个关键节点有状态、日志和指标。

---

## 5. 端到端目标链路（优化后）

1. 教师提交出题请求（结构化参数 + 可选追问）。
2. 系统创建生成任务（PENDING），立即返回 taskId。
3. 后台组装上下文（system + 最近N轮 + 当前请求）并调用模型。
4. 结果按 JSON Schema 解析，不通过则自动修复/重试。
5. 通过质量闸门后，写入 QuestionBank，并生成会话可读内容。
6. 教师在“会话视图”可继续追问；在“题库视图”可筛选复用。
7. 组卷/分发时引用题目ID集合，不再依赖大段自由文本。
8. 学生作答后回流统计（错误率、知识点薄弱），反哺后续出题参数推荐。

## 6. 详细优化设计

## 6.1 对话与上下文治理

### 6.1.1 上下文构建策略
- system：教师助手角色、教学边界、输出约束。
- history：最近 6~10 轮（按 token 预算裁剪）。
- summary：超长会话采用摘要压缩保留长期意图。
- user：当前请求。

### 6.1.2 上下文风控
- 过滤无关/越权上下文片段。
- 引入“指令优先级”：系统指令 > 业务规则 > 用户要求。

## 6.2 Prompt 模板与结构化输出

### 6.2.1 模板库
- 按题型拆模板：选择/填空/判断/简答/解答。
- 按学段扩展约束：小学/中学/大学不同表达与难度。

### 6.2.2 统一 JSON Schema（核心）
输出最小单元建议：
- stem（题干）
- type（题型）
- options（可选）
- answer（标准答案）
- analysis（解析）
- difficulty（难度）
- knowledgePoints（知识点数组）
- score（分值）

### 6.2.3 失败处理
- 解析失败：进行一次“仅格式修复”调用。
- 仍失败：降级为 Markdown 展示并标记“未结构化”。

## 6.3 质量闸门（Quality Gate）

### 6.3.1 规则校验
- 必填字段完整性
- 选择题答案是否在选项内
- 分值合法性与总分校验

### 6.3.2 内容校验
- 敏感/违规内容过滤
- 重复题检测（同会话/同题库近似去重）
- 难度一致性粗检（规则 + 轻量模型判断）

### 6.3.3 处理策略
- ERROR：阻断入库，提示教师重试
- WARNING：允许入库但打标“待复核”

## 6.4 数据模型演进（建议）

### 6.4.1 新增核心表
- `generation_tasks`：任务状态、耗时、失败原因、重试次数
- `question_bank`：题目主表（结构化字段）
- `question_options`：客观题选项表
- `question_tags`、`question_tag_rel`：标签体系
- `question_versions`：题目版本历史（支持“改题”）

### 6.4.2 对现有表的最小调整
- `messages` 增加：`user_prompt`、`raw_model_response`、`structured_status`
- `assignments` 增加：`source_type`（message/bank）、`source_ref_id`

### 6.4.3 兼容策略
- 保留老字段与老链路。
- 新链路优先使用 `question_bank`，老数据逐步迁移。

## 6.5 接口设计（V2 草案）

- `POST /teacher/question/v2/tasks`：创建生成任务
- `GET /teacher/question/v2/tasks/{taskId}`：查询任务状态与结果
- `POST /teacher/question/v2/regenerate`：按反馈重生成
- `POST /teacher/question-bank`：保存题库
- `GET /teacher/question-bank`：检索题库
- `POST /teacher/assignments/from-bank`：按题目ID分发

## 6.6 前端交互优化

- “会话视图”与“题库视图”分离。
- 新增任务状态条：排队中/生成中/校验中/完成/失败。
- 失败可一键重试；警告可“继续保存并标记”。
- 导出从前端拼接改为后端导出服务（Word/PDF模板）。

## 6.7 出题闭环增强

- 作答结果回流到题目维度：正确率、平均得分、常错点。
- 下次出题时默认推荐：薄弱知识点 + 难度梯度。
- 教师可对题目标记“高质量/低质量”，进入反馈训练集。

## 6.8 安全与合规

- 输入侧：提示词注入防护（规则黑白名单）。
- 输出侧：内容审核（教育场景不当内容拦截）。
- 鉴权：会话、题库、课程数据均按 teacher_id 严格隔离。
- 审计：保留任务日志、模型响应摘要与操作轨迹。

## 6.9 性能与可观测性

### 6.9.1 性能
- 异步任务 + 线程池/队列隔离
- 热点模板缓存、词典缓存
- 大任务拆分并行（多套卷生成）

### 6.9.2 可观测
- 指标：QPS、成功率、P95时延、重试率、校验拦截率
- 日志：taskId 全链路追踪
- 告警：连续失败、超时积压、模型异常波动

## 7. 分阶段实施计划（仅设计）

## 7.1 Phase 1（P0，2~3周）
- 目标：稳定可控生成
- 内容：任务化异步、JSON结构化输出、基础质量校验、V2查询接口
- 验收：解析成功率与生成稳定性达到目标下限

## 7.2 Phase 2（P1，3~4周）
- 目标：题库沉淀与复用
- 内容：question_bank、检索筛选、从题库分发、版本管理
- 验收：题目复用率提升，分发链路脱离自由文本

## 7.3 Phase 3（P1/P2，2~3周）
- 目标：教学闭环增强
- 内容：作答统计回流、推荐参数、导出模板化
- 验收：教师二次编辑率下降、学生练习命中薄弱点提升

---

## 8. 风险与应对

- 模型输出不稳定：Schema + 修复调用 + 降级展示
- Token 成本上升：上下文裁剪 + 摘要压缩 + 缓存
- 旧链路迁移成本：双轨并行 + 分批切换 + 回滚开关
- 教师使用门槛：默认模板 + 智能推荐参数 + 引导文案

---

## 9. 验收清单（上线前）

- 功能：任务创建、状态查询、结构化结果展示、失败重试可用
- 数据：题目入库、版本记录、分发引用链正确
- 安全：鉴权隔离与敏感内容拦截通过
- 性能：压测达到P95目标
- 监控：核心指标看板与告警策略已配置

---

## 10. 结论

当前系统已具备“AI出题 + 会话管理 + 分发 + 学生AI批改”的基础能力，但核心仍是单轮文本生成与会话级存储，缺少结构化题目资产与质量治理。建议按“任务化 + 结构化 + 题库化 + 闭环化”四步推进，优先做 P0 稳定性与可控性升级，再扩展题库与智能推荐能力。
